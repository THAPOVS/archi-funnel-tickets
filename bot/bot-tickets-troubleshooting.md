## Bug 1: Comillas dobles en nombre de producto rompen los botones

**Síntoma:** algunos mensajes llegan al grupo con la foto del comprobante y el caption correcto, pero **sin botones Aprobar/Rechazar**. Make.com reporta la ejecución como SUCCESS (ops=2, sin error).

**Caso real:** orden #30248 con producto `"Aquí y ahora... un juego de niñas"` (las comillas son parte del nombre del producto en WC). El sendPhoto a Telegram retornaba 200 OK pero el `reply_markup` se descartaba.

**Causa raíz:** el módulo HTTP del Telegram sendPhoto en Make.com construye el body como **JSON raw string** con placeholders `{{1.items}}` que se interpolan literalmente. Cuando el campo `items` contiene comillas dobles, esas comillas terminan dentro del JSON sin escape, rompiendo la estructura del `reply_markup`. Telegram parsea lo que puede (la foto y el caption truncado) y descarta el resto.

**Fix aplicado (2026-05-08):**

1. **En el snippet WP (`snippet_id=37364`):** sanitizar comillas en `customer`, `email`, e `items` antes de enviar al webhook:
   ```php
   $name = str_replace('"', "'", $item->get_name());
   ```

2. **En el escenario A de Make.com** (capa de defensa adicional): el módulo Telegram sendPhoto reemplaza placeholders por versiones sanitizadas:
   ```
   {{replace(1.customer; "\""; "'")}}
   {{replace(1.items; "\""; "'")}}
   {{replace(1.email; "\""; "'")}}
   {{replace(1.phone; "\""; "'")}}
   ```

**Detección a futuro:** si una nueva orden viene con foto pero sin botones, chequear si el nombre del producto, customer, email o phone tiene `"`. La doble defensa (snippet + Make) debería evitarlo, pero si aparece otro caso, agregar más campos a la sanitización.

**Cómo testear:** crear un producto WP con un nombre que tenga comillas dobles (ej: `Test "comillas" doble`), hacer una compra, ver si los botones aparecen en el mensaje del bot.

## Bug 2: Operator del filter del router (`text:startswith` vs `text:startwith`)

**Síntoma:** el escenario B (callback) recibe el callback de Telegram (ops=1 sin error) pero ninguna rama del router se ejecuta. Ningún módulo posterior corre. Ningún botón funciona.

**Causa raíz:** Make.com usa el operator string **`text:startwith`** (singular, sin S al final). Si en el blueprint creado via API se usa `text:startswith` (plural, como sería intuitivo), Make NO lo reconoce y el filter nunca matchea. Make no reporta el error en los logs, solo "no rama matcheó".

**Fix aplicado:** en cada `filter.conditions[].o` del router, usar `text:startwith`:
```json
{"a": "{{1.callback_query.data}}", "b": "approve_", "o": "text:startwith"}
```

**Detección a futuro:** si un router en cualquier escenario futuro creado via API no matchea, sospechar del operator. La lista de operators correctos de Make: `text:equal`, `text:notequal`, `text:contain`, `text:notcontain`, `text:startwith`, `text:endwith`, `text:match` (regex). Singular en todos.

## Bug 3: Webhook custom creado via API necesita "Determine data structure" desde la UI

**Síntoma:** escenario con módulo `gateway:CustomWebHook` creado 100% via API blueprint da timeout consistente de 40 segundos. Error: `ModuleTimeoutError - The operation timed out`. "No bundles were generated by this operation."

**Causa raíz:** Make.com no inicializa el trigger del webhook si el módulo no pasó por el flow visual de "learning" desde la UI. El módulo no sabe cómo parsear el payload entrante, queda esperando indefinidamente.

**Fix aplicado:** una vez creado el escenario via API, abrir el editor en la UI, click sobre el módulo Webhook → "Redetermine data structure" → mandar payload sample con curl → click "Save". Después de eso el webhook procesa todos los payloads correctamente.

**Detección a futuro:** todo escenario nuevo con webhook custom creado via API tiene que pasar por este paso manual UNA vez antes de funcionar.

## Bug 4: HTTP module legacy requiere campos obligatorios que el blueprint API no rellena

**Síntoma:** módulo HTTP creado via API blueprint da `BundleValidationError - Validation failed for 2 parameter(s)` en runtime. Errores: `Value not found in options in parameter 'method'` y `Missing value of required parameter 'followAllRedirects'`.

**Causa raíz:** el módulo `http:ActionSendData` versión 3 (HTTP legacy) requiere:
- `method` en lowercase: `'get'`, `'post'`, `'put'`, etc. (NO mayúscula como en el HTTP estándar)
- `followAllRedirects: false` (boolean obligatorio)

Cuando se crea via API blueprint estos campos no quedan bien seteados.

**Fix aplicado:** en cada módulo HTTP del blueprint, asegurar:
```python
mapper["method"] = "get"  # o post/put en lowercase
mapper["followAllRedirects"] = False
```

## Bug 5b: Telegram sendPhoto rechaza PDFs silenciosamente (causa raíz de mensajes "no llegaron")

**Síntoma:** clientes que suben el comprobante en formato PDF (Mercado Pago descarga PDF, algunos bancos también) no aparecen al grupo Telegram. Make.com reporta ops=2 sin error. Aprobadores se enteran del problema porque el equipo lo flaguea ("estas no llegaron").

**Caso real (2026-05-11):** equipo del Archi flagueó 6 órdenes que no aparecieron en el grupo: #37441 Gabriela Bianchi, #37404 Laura Andrade, #37349 Gonzalo Rivarola, #37347 gonzalo segal, #37345 Federico Scalise, #37341 Luciana Cabezas. Todas con comprobante PDF. Las aprobó manualmente desde WP admin el user "Archi".

**Causa raíz:**
1. PeproDev permite subir `image/jpeg, image/png, application/pdf` como comprobante.
2. El bot usa `https://api.telegram.org/bot<TOKEN>/sendPhoto` que **solo acepta imágenes** (JPG, PNG, GIF, WebP). NO acepta PDF.
3. Telegram retorna `{"ok":false,"error_code":400,"description":"Bad Request: wrong type of the web page content"}`.
4. **El módulo HTTP de Make tenía `evaluateAllErrors: false`** por default, así que cualquier response no-2xx era tratado como "OK". Make registraba ops=2 success y el escenario terminaba "limpio" sin alerta.

**Fix aplicado (2026-05-11):**

1. **Router en el escenario A** después del webhook. Dos ramas según extensión del `receipt_url`:
   - Filter `receipt_url ends with ".pdf"` → módulo HTTP a `/sendDocument` (acepta cualquier MIME).
   - Filter `receipt_url NOT ends with ".pdf"` → módulo HTTP a `/sendPhoto` (inline preview).

2. **`evaluateAllErrors: true` + `handleErrors: true`** en ambos módulos HTTP. Si Telegram devuelve no-2xx, Make ahora reporta error visible en los logs en vez de silenciar.

**Comportamiento esperado:**
- Comprobante en JPG/PNG → llega como foto con preview inline + botones.
- Comprobante en PDF → llega como adjunto con icono de PDF (tap para abrir) + caption + botones.

**Detección a futuro:** si una orden no llega al grupo, chequear logs de Make scenario A. Con `evaluateAllErrors=true`, cualquier error de Telegram queda registrado. Si el escenario termina con error, hay que diagnosticar (rate limit, formato no soportado, URL pública no accesible, etc.).

**Mejora 2026-05-11: preview inline de PDFs.** WP/ImageMagick genera thumbnails de PDFs uploadeados automáticamente con patrón `<filename>-pdf.jpg` (verificado en archibrazo.org). El escenario A ahora usa un módulo `util:SetVariable2` antes del router que calcula `photo_url = replace(receipt_url; ".pdf"; "-pdf.jpg")`. Ambas ramas del router (PDF e imagen) usan `sendPhoto` con `{{2.photo_url}}` como photo. Resultado: PDFs se ven con preview inline del thumbnail (primera página) + caption + botones + link al PDF original al final del caption (para que el coope pueda descargar el original si necesita ver páginas adicionales). Imágenes JPG/PNG: replace no afecta, se mantienen como antes.

**Slug correcto del módulo SetVariable en blueprints API:** `util:SetVariable2` (mayúsculas exactas). Los slugs `util:setVariable`, `tools:SetVariable`, `builtin:setVariable` y similares NO existen y devuelven "Module not found". Mapper requerido:
```json
{
  "name": "photo_url",
  "scope": "roundtrip",
  "value": "{{replace(1.receipt_url; \".pdf\"; \"-pdf.jpg\")}}"
}
```
El `scope: "roundtrip"` indica que la variable vive solo durante esa ejecución. Otros scopes posibles: `execution` (todo el run), `scenario` (persistente entre runs).

## Bug 5: WP REST API `/wp-json/wp/v2/media/` no acepta WC API key

**Síntoma:** llamada a `/wp-json/wp/v2/media/<attachment_id>` con Basic Auth de WC consumer_key/secret retorna 401 "invalid_username".

**Causa raíz:** las claves WC REST API (consumer_key/consumer_secret) **solo funcionan en endpoints `/wp-json/wc/v3/*`**. Para endpoints de WP Core (`/wp-json/wp/v2/*`) hay que usar:
- Cookie auth + nonce (sirve solo para usuarios logueados via browser)
- Application Passwords (Profile WP → Application Passwords → Add)

**Fix aplicado:** en el snippet WP server-side ya tiene acceso al filesystem y puede llamar `wp_get_attachment_url($id)` directamente, sin pasar por la WP REST API. El payload enviado al webhook incluye `receipt_url` ya resuelta. El escenario A no necesita hacer ningún GET adicional.

## Decisiones de diseño importantes

### Idempotencia post-acción

Cuando un coope aprieta Aprobar o Rechazar, el escenario B llama `editMessageReplyMarkup` para **eliminar los botones del mensaje original**. Esto previene que otro coope (o el mismo) apriete dos veces y cause inconsistencia (ej: aprobar, después rechazar, el cliente queda con tickets generados pero orden marcada como rechazada).

### Sincronización de meta `receipt_upload_status`

Cuando el bot aprueba/rechaza vía API, además de cambiar el estado general de la orden (`completed` o `receipt-rejected`) también actualiza el meta `receipt_upload_status` (`approved` o `rejected`). Esto evita que en el WP admin se vea "Comprobante Aprobado: pending" mientras la orden está completed (inconsistencia visual confusa).

### Validación humana visible (formato del caption)

El caption del mensaje al grupo incluye:
- `customer`, `email`, `phone` para identificar al comprador.
- `items` formateado como `nombre x[qty] ($X c/u = $Y)` para que el coope pueda **multiplicar mentalmente y validar contra el monto del comprobante**.
- `total` general de la orden.
- Link al admin de WP por si hace falta entrar a verificar más datos.

### Backlog histórico (órdenes muy viejas) - silent close masivo

Las órdenes con eventos ya pasados no conviene aprobarlas (FooEvents les manda tickets PDF para algo ya hecho) ni rechazarlas (el cliente recibe mail negativo confuso). Lo prolijo es cancelarlas en silencio.

**Setup:** snippet WPCode "Silent close - bulk archive" (snippet_id 37397) ya creado y activo. Silencia todos los emails de WC + tickets/emails de FooEvents para órdenes con meta `_bot_silent_close=1`. Es inocuo dejarlo activo permanentemente porque solo afecta órdenes con esa meta seteada.

**Procedimiento bulk via WC API:**

```javascript
// Ejecutar desde browser logueado al WP admin
const WC_AUTH = 'Basic <base64 ck:cs>';
const CUTOFF = '2026-04-15';  // ajustar según necesidad
const lists = [];
for (const st of ['receipt-approval', 'receipt-upload']) {
  const r = await fetch(`/wp-json/wc/v3/orders?status=${st}&per_page=100&orderby=date&order=asc`, {
    headers: {Authorization: WC_AUTH}, credentials: 'omit'
  });
  const arr = await r.json();
  arr.forEach(o => {
    if (o.date_created < CUTOFF) lists.push({id: o.id, date: o.date_created});
  });
}

for (const o of lists) {
  await fetch('/wp-json/wc/v3/orders/' + o.id, {
    method: 'PUT',
    headers: {Authorization: WC_AUTH, 'Content-Type': 'application/json'},
    body: JSON.stringify({
      status: 'cancelled',
      meta_data: [
        {key: '_bot_silent_close', value: '1'},
        {key: 'receipt_upload_status', value: 'rejected'}
      ]
    })
  });
  await fetch('/wp-json/wc/v3/orders/' + o.id + '/notes', {
    method: 'POST',
    headers: {Authorization: WC_AUTH, 'Content-Type': 'application/json'},
    body: JSON.stringify({
      note: 'Cancelada masivamente por backlog historico (' + o.date.substring(0,10) + ').',
      customer_note: false
    })
  });
  await new Promise(r => setTimeout(r, 400));
}
```

**Ejecutado 2026-05-08:** 73 de 75 órdenes pre-2026-04 canceladas silenciosamente. Las 2 que fallaron (#30740, #30741) eran órdenes vacías con 0 line_items (rotas en origen). No requieren cleanup.

**Importante:** la meta `_bot_silent_close=1` queda en la orden permanentemente. Si en el futuro cambia de estado por cualquier motivo, los emails siguen silenciados. Es intencional. Si por algún motivo se quiere rehabilitar emails para esa orden, borrar la meta o cambiar a `_bot_silent_close=0`.

## Bug 7 (Gemini auto-evaluación — en implementación 2026-05-17)

> Plan completo en `~/.claude/plans/donde-armamos-el-bot-noble-seal.md`. Credenciales en `secrets/bot-tickets-credentials.md`.

### Lessons confirmados con test e2e 2026-05-17

- **Modelo: `gemini-2.5-flash`, NO `gemini-2.0-flash`.** El free tier de `gemini-2.0-flash` ya no aplica para proyectos creados en 2026. Síntoma: HTTP 429 con `limit: 0` para todos los rate limits. Solución: usar `gemini-2.5-flash` (mejor para vision, mismo costo, free tier activo).
- **CRÍTICO: `thinkingConfig: {thinkingBudget: 0}` obligatorio.** Sin esto, gemini-2.5-flash usa ~500 tokens "thinking" antes de generar el JSON, y con `maxOutputTokens: 500` el response queda truncado a media palabra. Síntoma: JSON parse error con string `"monto_detectado` (cortado). Solución: agregar `"thinkingConfig":{"thinkingBudget":0}` en `generationConfig`. Para esta tarea (OCR+comparación) no se necesita razonamiento profundo.
- **Cuenta `joaquin@paretoppc.com` (Workspace) no funciona.** Las cuentas Google Workspace tienen policies de organización que deshabilitan free tier de Gemini API a pesar de mostrar "Free tier" en la UI. Usar cuenta personal `@gmail.com` (ej. `archibrazo@gmail.com`).
- **Path del response.** Confirmado: `{{4.candidates[1].content.parts[1].text}}` (1-indexed en Make). El text es un JSON string que necesita json:ParseJSON adicional con `type: "flat"`.
- **Token cost real:** ~600 tokens/comprobante (288 prompt text + 258 image + ~80 output). En tier pago serían ~$0.00009/comprobante. En free tier: $0. Para 100 órdenes/semana = $0.04/mes si nos pasáramos a tier pago.
- **Latencia:** <5 seg sin thinking. Aceptable para webhook async.

### Cosas a vigilar cuando se active

- **`inline_data` vs `file_uri` para PDFs.** Para imágenes JPG/PNG funciona `inline_data` con base64 sin problema. Para PDFs el `photo_url` ya apunta al thumbnail `-pdf.jpg` (de la fix de Bug 5b), así que Gemini analiza el thumbnail (primera página) y no el PDF original. Si en algún caso se necesita analizar el PDF completo, usar `file_uri` con la URL del PDF.
- **Rate limit Gemini free tier 2.5-flash:** verificar específicamente en aistudio → Rate Limit. Para <100 órdenes/semana cualquier free tier alcanza.
- **Comprobante con monto sobrepagado.** Si el cliente paga $X+1000 (a veces pasa con redondeo o por error), Gemini va a marcar `monto_detectado != esperado` y va a bajar confidence. Es el comportamiento correcto: que vaya al grupo humano, no auto-aprobar pagos no-exactos. **Confirmado en test e2e:** con monto esperado $100 y comprobante real de $6600, Gemini devolvió `confidence: 40` y `motivo` explicando la discrepancia.
- **Falsos positivos.** Si Gemini auto-aprueba algo que no debía, revisar la nota privada de la orden (queda `🤖 Auto-aprobada por bot (Gemini X%): {motivo}`). Cuando aparezca el primero, hacer postmortem acá y subir threshold.

## Bug 8 (Detección de duplicados + doble-disparo del router — resuelto 2026-05-21)

### Síntoma

Una orden auto-aprobable disparaba la rama AUTO **y** una rama HUMAN al mismo tiempo (doble mensaje en el grupo: el aviso de auto-aprobada + el mensaje con botones). Y la detección de duplicados, en 5 intentos previos, rompía la auto-aprobación.

### Causa raíz

El `builtin:BasicRouter` de Make manda el bundle por **todas** las ramas cuyo filtro matchea, no por la primera. Las ramas HUMAN solo filtraban por extensión (`.pdf` / no `.pdf`), así que también matcheaban una orden ya auto-aprobada. Para que el router sea mutuamente excluyente, **cada rama tiene que tener un filtro AND-exhaustivo** que la separe de todas las demás.

Errores adicionales de los intentos previos de duplicados:
- El operador `text:notcontain` no funciona en Make (usar `text:notequal`, sí funciona y está validado).
- Marcar el resultado con la palabra `"OK"` choca con el `200 OK` del status HTTP cuando se usa `text:contain`.

### Solución (router 4 ramas AND-exclusivas)

El endpoint WP devuelve un campo limpio `dup: "SI"|"NO"` (string, igual que `auto_aprobable`). Filtros:
- **DUPLICADO:** `dup == SI`
- **AUTO:** `dup != SI` AND `auto_aprobable == SI`
- **HUMAN PDF:** `dup != SI` AND `auto_aprobable != SI` AND `receipt_url endwith .pdf`
- **HUMAN Imagen:** `dup != SI` AND `auto_aprobable != SI` AND `receipt_url notendwith .pdf`

Exactamente una rama matchea para cualquier input. Nunca usar `text:contain`/`text:notcontain` en filtros del router: solo `text:equal`/`text:notequal`/`text:endwith`/`text:notendwith`.

### Detección de duplicados (snippet WP 37832 + módulos 6/7)

- El snippet 37832 compara `order_id` **server-side**: devuelve `dup:SI` solo si el número ya está registrado para una orden **distinta**. La misma orden re-procesada (re-disparo del webhook) devuelve `dup:NO` → no se auto-rechaza sola.
- Módulo 6 (HTTP GET): `?num={{5.numero_operacion}}&order_id={{1.order_id}}&key=...`, `parseResponse:true` → se lee `{{6.data.dup}}`.
- Módulo 7 (HTTP POST): registra el número. El snippet hace no-op si ya estaba registrado o si `num` viene vacío. Corre para toda orden, antes del router.
- Si Gemini no lee el número (`numero_operacion` vacío) → endpoint devuelve `dup:NO` → la orden sigue el flujo normal. Falla seguro.

### Validado e2e 2026-05-21

Orden test #37732 (total 8800) procesada con el comprobante real de Ivan (operación 159284573581, ya registrada a la orden #37665): aunque el monto coincidía y era auto-aprobable, fue a la rama DUPLICADO, quedó en `receipt-rejected` con nota privada del bot, sin doble-disparo, sin contaminar el registro. La rama AUTO no se dispara para un duplicado porque exige `dup != SI`.
